🏗️ MyOwnAnthropicServer — what the Messages API server actually does

Design your own minimal POST /v1/messages to see every responsibility Anthropic handles for you

THE IDEA The model is one line; the rest is the server

The best way to understand what the Anthropic server does is to sketch your own. When you do, the punchline jumps out: calling the LLM is a single line in the middle — auth, rate limits, validation, prompt assembly, caching, safety, response shaping, billing, and streaming are all the server's job, before and after that line. This page designs that shell (not the transformer — that stays a black box) so you can see the responsibilities.

🎯 This is the synthesis of the whole D5/D4 error & cost material: who catches what, cost, caching, and stop-reasons — all visible as stages of one request.

DO YOU ALWAYS NEED THE SERVER? Three deployment shapes

LangGraph doesn't remove the server. LangGraph / LangChain / CrewAI / LlamaIndex are client-side orchestrators — they decide what to call, not where the model runs. LangGraph → OpenAI is shape A (it calls OpenAI's API server over HTTP, which still does auth/limits/validation/safety). It only becomes "direct" when you point it at a local Ollama (B) or an in-process model (C).
The punchline: the 13 stages don't disappear in B and C — they either become your job (self-hosted, multi-user) or become unnecessary (single-user local: no auth, no rate limits, no billing). That's exactly why the hosted API exists — you rent those responsibilities instead of building them. MyOwnAnthropicServer is what you'd have to write to turn shape C back into shape A.

THE MINIMAL SERVER ~40 lines — one function per responsibility

# MyOwnAnthropicServer — a teaching shell for POST /v1/messages.
# The transformer is a black box (model.generate). Everything AROUND it is the server's job.

def handle_messages(http_request):
    # 1 · TRANSPORT — parse the HTTP body
    try:    body = json.loads(http_request.body)
    except: return err(400, "invalid_request_error", "malformed JSON")

    # 2 · AUTH — who is this? (x-api-key, anthropic-version)  → 401 / 403
    org = authenticate(http_request.headers.get("x-api-key"))
    if not org:                    return err(401, "authentication_error", "invalid key")
    if not org.can_use(body.get("model")): return err(403, "permission_error", "no access")

    # 3 · RATE LIMIT / QUOTA — token buckets per org (RPM/ITPM/OTPM)  → 429 + Retry-After
    if not rate_limiter.allow(org, body): return err(429, "rate_limit_error", retry_after=2)

    # 4 · SIZE — payload cap  → 413
    if too_large(body):        return err(413, "request_too_large", "body too large")

    # 5 · VALIDATE — the big 400 layer, BEFORE the model ever runs
    bad = validate(body)          # required fields · role alternation · tool_use<->tool_result
    if bad: return err(400, "invalid_request_error", bad)  # · max_tokens<=limit · no temperature/budget_tokens

    # 6 · ROUTE — resolve model id -> engine; check the context window  → 404 if unknown
    model = resolve_model(body["model"])

    # 7 · ASSEMBLE PROMPT — render tools -> system -> messages into model format; count tokens
    prompt, n_in = render(body)     # enforce max input (context-window-exceeded is a 400)

    # 8 · PROMPT CACHE — hash the stable prefix; reuse KV on a hit (skip prefill)
    kv, cache_read, cache_new = prompt_cache.lookup(prompt, body)

    # 9 · INFER — the ONE line that is "the model" (prefill + autoregressive decode)
    out = model.generate(prompt, kv,
        max_tokens      = body["max_tokens"],          # → stop_reason "max_tokens"
        stop_sequences  = body.get("stop_sequences"),  # → "stop_sequence"
        thinking        = body.get("thinking"),        # adaptive reasoning (billed as output)
        tools           = body.get("tools"),           # constrained decode → "tool_use"
        output_config   = body.get("output_config"))   # schema-constrained JSON

    # 10 · SAFETY — classifiers can override the output  → stop_reason "refusal"
    if safety.blocks(body, out): out = refusal()

    # 11 · RESPONSE — content blocks (text/thinking/tool_use) + stop_reason + usage
    resp = build(out, stop_reason=out.stop_reason,
        usage=usage(n_in, out.n_out, cache_read, cache_new))  # the 4 usage fields

    # 12 · METER + LOG — record billing; attach request-id for tracing
    meter(org, resp.usage); log(http_request.id, org, resp)

    # 13 · RETURN — one JSON body, or a stream of SSE events if body["stream"]
    return sse_stream(resp) if body.get("stream") else json_response(resp)
Read stages 1–8, then 10–13, and notice: only stage 9 is "the model." Everything else is the responsibility you get for free — and exactly what an exam question about "where does X happen?" is testing.

ZOOM: STAGE 9 what "call the LLM" actually hides

The one line model.generate() is itself a small pipeline — worth knowing for the exam even though you'd never build it:

  • Prefill — process the whole prompt in one pass, building the KV cache (this is what prompt caching reuses → the ~0.1× cache-read).
  • Decode — generate output tokens one at a time, autoregressively (this is why output is slow & costs 5×).
  • Stop conditions — halt on natural end (end_turn), the max_tokens cap, a stop_sequence, or a tool call (tool_use) → the server stamps the matching stop_reason.
  • Constrained decoding — for tools/output_config.format, sampling is restricted to tokens valid under the schema/grammar (this is why structured outputs can't produce malformed JSON — the shape is guaranteed at decode time).
  • Thinking — extra reasoning tokens generated before the answer, billed as output.

SCOPE what we deliberately DON'T build

Not in scope: the transformer itself (attention, weights, the KV cache internals). That's the model, not the server — and reimplementing it teaches you nothing about the API's responsibilities. We keep it as an opaque model.generate(), which is exactly the boundary Anthropic exposes to you.

CLIENT vs SERVER the responsibility split (and why statelessness)

Your client doesThe server does
Build & resend the full message history every turnTreat each request as stateless — no session memory
Loop on stop_reason, run tools, return tool_resultDecode the tool call, stamp tool_use, validate the returned pairing
Retry 429/5xx with backoff; set idempotencyEnforce limits, return typed errors, meter usage
Decide model, max_tokens, caching breakpointsRoute, cache, assemble, infer, shape the response
Statelessness is a server responsibility you feel as a client cost: because the server keeps no session, you resend the whole history — which is why long chats get expensive (every past token re-billed). Build this shell and that trade-off stops being abstract.

LEVEL IT UP make the shell actually run

  • Wrap a local model → replace model.generate() with an Ollama call and serve POST /v1/messages with FastAPI — a toy Messages-API-compatible server you can point the Anthropic SDK at (base_url).
  • Implement one real stage → e.g. a genuine validate() that reproduces the 400 rules, or a real token-bucket rate limiter.
  • Add streaming → emit the SSE event sequence (message_start → content_block_delta → message_stop).
  • Add a prefix cache → hash the stable prefix and skip re-work on a hit; watch your own cache_read counter.
Doing even one stage for real turns the abstract "the API handles it" into muscle memory — which is exactly the level the CCA exam probes.
