🧰 GenericAnthropicMessageClient — one robust client, every case

A design-patterns study: handle every request/response permutation of the Messages API behind one interface

THE IDEA The client-side twin of the server

If MyOwnAnthropicServer is everything the server does, this is everything the caller should do to be robust: handle all 6 stop reasons, the full error taxonomy, retries + backoff + idempotency, rate limits, token counting → max_tokens fit, context-bloat (trim / edit / compact), streaming, batch, structured outputs, tool loops, caching, and multi-backend routing — all behind one call. Your instinct is right: with code-to-interface design this collapses into a small set of pluggable pieces.

🎯 The lesson isn't "reinvent the SDK" — it's that a robust caller is a pipeline of policies, and each policy is an interface you can swap. That framing is exactly what the exam's reliability questions reward.

Tool Runner is the first-party attempt at the agentic-loop slice of exactly this idea.

THE DESIGN Facade + Middleware pipeline + Strategies

Three patterns do the heavy lifting: a Facade (one entry point), a Chain-of-Responsibility middleware pipeline (cross-cutting steps as an onion), and Strategy interfaces (each concern swappable).

THE INTERFACES code-to-interface (Python Protocols)

from typing import Protocol, Iterator

# ADAPTER — one interface over every backend (Anthropic / Databricks / Bedrock).
class Backend(Protocol):
    def create(self, req) -> "Response": ...
    def stream(self, req) -> Iterator["Event"]: ...
    def count_tokens(self, req) -> int: ...

# STRATEGY — retry (429/5xx only; NEVER 400), backoff + jitter, honor Retry-After.
class RetryPolicy(Protocol):
    def should_retry(self, err, attempt) -> bool: ...
    def delay(self, err, attempt) -> float: ...

# STRATEGY — fit the request into the window BEFORE sending.
class ContextPolicy(Protocol):
    def fit(self, req, n_tokens, window) -> "Request": ...   # trim / clear tool_results / compact / lower max_tokens

# STATE / dispatch — turn a stop_reason into the next Action.
class StopReasonHandler(Protocol):
    def handle(self, resp, ctx) -> "Action": ...
        # tool_use   -> run tools, append tool_result, CONTINUE
        # end_turn   -> DONE
        # max_tokens -> CONTINUE (or raise cap) — not done!
        # refusal    -> RETURN graceful decline
        # pause_turn -> RESUME (resend)          | stop_sequence -> DONE

# STRATEGY — text / structured(schema) / tool-args.
class OutputParser(Protocol):
    def parse(self, resp): ...

# CHAIN OF RESPONSIBILITY — each step wraps the next (onion).
class Middleware(Protocol):
    def __call__(self, req, nxt) -> "Result": ...

# FACADE — one entry point; composes a backend + a pipeline + policies.
class MessageClient:
    def __init__(self, backend: Backend, pipeline: list[Middleware],
                 retry: RetryPolicy, context: ContextPolicy,
                 router: StopReasonHandler, parser: OutputParser): ...
    def send(self, req) -> "Result": ...        # runs the pipeline; loops on the router until DONE
    def stream(self, req) -> Iterator: ...       # same policies, streamed
    def submit_batch(self, reqs) -> "BatchHandle": ...  # same request shape, async ~50% cost
Every hard case from the matrix is now a named interface with one job. Swapping "sliding window" for "compaction" is a one-line policy change, not a rewrite — that's the whole point of coding to interfaces.

THE HEART stop_reason as a state machine

The messiest part — the loop — becomes a tiny dispatch once StopReasonHandler owns it:

stateDiagram-v2 [*] --> Call Call --> Route: 200 + stop_reason Route --> RunTools: tool_use RunTools --> Call: append tool_result Route --> Continue: max_tokens (truncated) Continue --> Call Route --> Resume: pause_turn Resume --> Call Route --> Done: end_turn / stop_sequence Route --> Decline: refusal Done --> [*] Decline --> [*]
The exam trap this design kills: forgetting a stop_reason. A dispatch table with a required case for all six (and a catch-all) makes "silently loop on max_tokens" impossible by construction.

VS TOOL RUNNER what each covers

🛠️ Tool Runner (first-party)

  • Automates the tool-use loop for tools you define
  • Per-turn hooks (approval, interception, result edit, retries)
  • Streaming, compaction support
  • Zero loop code — great for production agents

🧰 GenericMessageClient (your Facade)

  • Superset scope: also batch, context-fit, multi-backend, fallbacks, output-parse strategy
  • You own the policy interfaces — house rules in one place
  • More to build & maintain
  • Best as a thin wrapper that uses the SDK/Tool Runner inside
Recommended reality: don't reimplement retries/streaming the SDK already does — compose. Let the Facade delegate the tool loop to Tool Runner, retries to the SDK, and add only the policies they don't cover (context-fit, batch routing, fallbacks, budget). Build-your-own is for learning the surface; ship a thin wrapper.

LEVEL IT UP

  • Build the vertical slice → Facade + a RetryPolicy + a StopReasonHandler over the real SDK. That alone is a great portfolio piece.
  • Add a ContextPolicy → plug in sliding-window, then swap to context-editing, then compaction — same interface.
  • Add a BackendAdapter → one for Anthropic-direct, one for Databricks Model Serving (base_url) — prove the Adapter.
  • Add fallbacks → a chain: Opus → Sonnet → cached — Chain of Responsibility in action.
  • Wrap Tool Runner → let the Facade delegate the loop; you only own the policies around it.
