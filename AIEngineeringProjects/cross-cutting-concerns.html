🧩 Cross-Cutting Concerns — pluggable ports

Keep logging, tracing, cost, security & more OUT of the core — behind interfaces, with swappable adapters

THE IDEA A concern is cross-cutting when it touches every call but isn't the logic

Logging, tracing, metrics, cost, redaction, caching, rate limits, audit — none of these are "call the model," yet every request needs them. If you scatter them through the core you get an unmaintainable, untestable mess. The fix is the same principle you liked: code to interface. The core depends only on a port (e.g. LogSink); where it physically logs is a swappable adapter (stdout / CloudWatch / Azure Monitor). That's ports-and-adapters (hexagonal) architecture + dependency injection, applied to GenericAnthropicMessageClient.

🎯 Two rules make this work: (1) the core imports interfaces, never concrete sinks (Dependency-Inversion). (2) The concrete adapters are wired in one place — the composition root. Swapping "log to file" for "log to CloudWatch" is then a one-line change, and tests inject fakes.

THE PORTS interfaces the core depends on (Python Protocols)

from typing import Protocol, Any

class LogSink(Protocol):
    def emit(self, record: dict) -> None: ...        # adapter decides WHERE (stdout/CloudWatch/...)

class Tracer(Protocol):
    def span(self, name: str, **attrs): ...             # context manager; nests per loop step

class MetricSink(Protocol):
    def incr(self, name: str, value: float = 1, **tags): ...
    def observe(self, name: str, value: float, **tags): ...

class Budget(Protocol):
    def check(self, est_tokens: int) -> bool: ...        # deny before the call if over
    def record(self, usage) -> None: ...              # debit after

class Redactor(Protocol):
    def scrub(self, obj: Any) -> Any: ...              # PII/secrets out BEFORE anything logs it

class SecretProvider(Protocol):
    def get(self, name: str) -> str: ...

class CacheStore(Protocol):
    def get(self, key: str): ...
    def put(self, key: str, value, ttl: int) -> None: ...

class AuditSink(Protocol):
    def record(self, event: dict) -> None: ...          # append-only, immutable

class RateLimiter(Protocol):
    def acquire(self, cost: int) -> None: ...          # block/throttle client-side

class GuardrailPolicy(Protocol):
    def check_input(self, req) -> None: ...
    def check_output(self, resp) -> None: ...

HOW THEY WEAVE IN the middleware order matters

Cross-cutting behaviour applies as middleware around the core call (Decorator/Chain-of-Responsibility). Order is a design decision, not an accident:

flowchart TB R["request"] --> RD["Redact (scrub secrets/PII)"] --> BG["Budget.check"] --> RL["RateLimiter.acquire"] RL --> CH["Cache.get (hit? short-circuit)"] --> GI["Guardrail.check_input"] --> TR["Tracer.span + Log start"] TR --> CALL["→ Backend.call (+ Retry) ←"]:::c --> GO["Guardrail.check_output"] --> BR["Budget.record + Metrics"] BR --> AU["Audit.record"] --> CP["Cache.put"] --> LOG["Log end"] --> OUT["result"] classDef c fill:#dcfce7,stroke:#15803d,color:#14532d;
Ordering traps: Redact must run first — before anything logs or traces the payload, or you leak PII into your telemetry. Budget/RateLimit before the call (no point paying to reject later). Cache before Guardrail-input? depends — cache the validated request. Getting the order right is half the design.

THE COMPOSITION ROOT wire concrete adapters in ONE place

# composition_root.py — the ONLY file that names concrete adapters (dependency injection)
client = MessageClient(
    backend = DatabricksBackend(host=cfg.host, token=secrets.get("DBX_TOKEN")),  # Adapter
    ports   = Ports(
        log    = CloudWatchLog()      if prod else StdoutLog(),        # swap WHERE we log, 1 line
        tracer = LangfuseTracer()     if prod else PhoenixTracer(),
        metric = PrometheusMetrics(),
        budget = RedisBudget(cap_usd=cfg.daily_cap),
        redact = PIIRedactor(),
        cache  = RedisCache()          if prod else InMemoryCache(),
        audit  = DeltaAudit()          if prod else JsonlAudit(),
        limiter= TokenBucket(rpm=cfg.rpm),
        guard  = CompositeGuardrail([InjectionFilter(), Moderation()]),
    ),
)
# Tests: pass FakeLog(), FakeBudget(), InMemoryCache() — no network, deterministic.
This is the payoff: the core logic is written once and never mentions CloudWatch, Redis, or Langfuse. Dev vs prod, local vs Azure, real vs fake — all decided here. That's testability and portability by construction.

LEVEL IT UP

  • Build two adapters per port → e.g. StdoutLog + JsonlAudit locally, then a cloud one; prove the swap.
  • Order the middleware deliberately → put Redact first and verify no PII reaches the log sink.
  • Inject fakes in tests → assert the client logs/meters/audits correctly with zero network.
  • Add a kill switch → a FlagSource that disables a model/tool instantly (a guardrail in practice).
  • Fold into GenericMessageClient → these ports are its constructor dependencies.
