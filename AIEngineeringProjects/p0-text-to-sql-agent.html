🔁 P0 · Self-Correcting Text-to-SQL Agent

The smallest pure agent · one loop, every core concept · runs free on your laptop

THE PROJECT One line, and why it's the teaching agent

Ask a question in English → the agent writes SQL, runs it, and if it errors, reads the database error and fixes the query, looping until it works — then answers. It's the smallest thing that is genuinely agentic: you cannot know the SQL is correct without running it, so the decide → act → observe → adjust loop is unavoidable (not a workflow you could hardcode). Read-only, ~100 lines, one dataset — so you learn the loop and the tool contract with nothing else in the way.

🎯 Everything bigger (the DQ agent, the UC-migration agent, RCA agents) is this loop + more tools + a verification step. Master it here.

1 · CONCEPTUAL DESIGN The logic, no infrastructure

Conceptual design answers "what are the moving parts and how do they interact?" — independent of where it runs. There are only four logical roles plus a loop:

Logical roleResponsibility
Agent / OrchestratorOwns the loop: prompt the model, execute its tool calls, feed results back, decide when to stop
LLM (reasoner)Turns question + schema (+ prior errors) into a SQL query; later, turns rows into a plain answer
Toolsget_schema() and run_sql(query) — the agent's only ways to touch the world
Data (SQL engine)Executes the query, returns rows or an error message (the crucial feedback signal)

The loop, as a sequence

sequenceDiagram actor U as User participant AG as Agent (loop) participant M as LLM participant DB as SQL engine U->>AG: "How many EU orders last quarter?" AG->>DB: get_schema() DB-->>AG: tables & columns loop until success or max_tries AG->>M: question + schema (+ last error) M-->>AG: SQL query AG->>DB: run_sql(query) alt query errors DB-->>AG: error message Note over AG,M: feed the error back → model fixes it else query runs DB-->>AG: rows AG->>M: rows → summarize M-->>AG: plain-English answer AG-->>U: answer (+ the SQL) end end

✅ What makes this an agent

  • The next action depends on the observed result (the error), not a fixed script
  • The model, not your code, decides the fix
  • There's a real feedback loop and a terminal condition

⚠️ Design decisions to make now

  • Loop termination: cap at N tries (D1) — never loop forever
  • Tool contract: how errors are returned to the model (D4)
  • Guardrails: read-only, row limit, timeout (built in, not prompted)

3 · CROSS-CUTTING CONCERNS The stuff that decides if it's real

ConcernWhat it means for this agentLevers / practices
💰 CostThe schema is re-sent on every retry; each attempt = input tokens + output SQL. Retries multiply it.Cache the schema prompt · cap max-tries (2–4) · use a small SQL-tuned model · trim schema to relevant tables
📊 ObservabilityYou need to see the agent think: each iteration's SQL, the error, the fix, tokens, latency, #retries, success/fail.Trace every loop step (Phoenix/Langfuse/MLflow) · a run-id per request · dashboards on success-rate & avg-retries
🔐 SecurityUser text is untrusted (prompt injection) and SQL can be destructive. "Please DROP TABLE" must be impossible, not just discouraged.Enforce at the DB layer: read-only role, no DDL/DML, allow-listed schemas · statement timeout · LIMIT · redact PII in logs · never trust the prompt for safety
📝 LoggingReproduce and debug any run; prove what happened; don't leak data.Structured JSON logs (prompt, SQL, result-shape, tokens) · correlation id · scrub secrets/PII · log levels
🛡️ ReliabilityModel errors, DB timeouts, no-valid-SQL-found.Loop cap + graceful "I couldn't answer" · retry model calls with backoff · timeouts on every tool
✅ Evals"Did it work?" is measurable here — the query runs and returns the right answer, or not.Execution accuracy (does it run?) + result match vs gold queries + avg #self-corrections · run on every prompt/model change
The #1 real-world failure: relying on the prompt ("only write SELECT queries") for safety. A read-only database grant enforces it even if the model is tricked. Security lives in the tool/permission layer, never in the prompt.

5 · BUILD IT FREE & LOCAL $0, offline, on your machine

What we'll actually run — every piece free and local:

🧠 Model — Ollama with an open SQL/coder model (qwen2.5-coder, llama3.1, or sqlcoder). Free, offline.
🗄️ Data — DuckDB (embedded, zero-setup, reads CSV/Parquet).
🔁 Agent — plain Python (no framework — you see every step).
📊 Tracing — Arize Phoenix (pip install, runs locally).
📦 Containers — optional Docker Compose to practice the deployment shape.

Since this repo is Claude-focused: in a real build you'd point the same code at Claude via the Anthropic API (or Bedrock/Vertex/Foundry). The client is swappable — only the endpoint changes. Free-tier hosted alternatives to Ollama: Google AI Studio, Groq, or OpenRouter :free models.

The whole agent (minimal, real)

# pip install duckdb requests   ·   ollama pull qwen2.5-coder:7b
import duckdb, requests, json

# --- Data: open READ-ONLY so DROP/DELETE are impossible at the engine layer ---
con = duckdb.connect("demo.duckdb", read_only=True)

def get_schema():
    return con.sql("""SELECT table_name, column_name, data_type
                      FROM information_schema.columns ORDER BY table_name""").df().to_string()

def run_sql(q):                       # the tool the model calls
    return con.sql(f"SELECT * FROM ({q}) LIMIT 100").df()   # guardrail: cap rows

def ask(prompt):                        # call the local model (Ollama)
    r = requests.post("http://localhost:11434/api/generate",
                      json={"model":"qwen2.5-coder:7b", "prompt":prompt, "stream":False})
    return r.json()["response"].strip()

def agent(question, max_tries=4):        # THE LOOP — this is the whole agent
    schema, err = get_schema(), ""
    for attempt in range(max_tries):
        sql = ask(f"DuckDB schema:\n{schema}\nQuestion: {question}\n"
                  f"{err}Return ONLY a SELECT query, no prose.")
        try:
            rows = run_sql(sql)                        # act
            return ask(f"Q: {question}\nRows:\n{rows}\nAnswer in one sentence.")  # success → stop
        except Exception as e:
            err = f"Your last query failed with: {e}\nFix it. "    # observe → feed back → loop
    return "I couldn't answer that reliably."          # graceful termination

print(agent("How many orders were placed in 2024?"))
That's it — the for loop with the try/except feedback is the agent. Everything else in this page is what turns these 25 lines into something you'd trust in production.

6 · PRODUCTION FAILURE POINTS

Prompt injection in the question → user embeds "ignore instructions, DROP TABLE". Mitigation: read-only DB grant + allow-listed schemas — safety at the engine, not the prompt.
Runaway / expensive queries → a cross-join scans billions of rows. Mitigation: statement timeout, LIMIT, and (on real warehouses) a cost/bytes-scanned cap.
Confidently wrong SQL that runs → wrong join, wrong filter — returns a plausible but wrong number. Mitigation: evals vs gold queries; show the SQL to the user; ask the model to self-check against the question.
Retry storm → it never finds valid SQL and loops/costs forever. Mitigation: hard max-tries + graceful failure (loop termination, D1).
Schema too big for context → hundreds of tables blow the window/cost. Mitigation: retrieve only relevant tables (RAG over the schema) + cache the stable part.
