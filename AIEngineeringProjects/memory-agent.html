🧠 Second-Brain Assistant — Tiered Memory & Context Engineering

A toy that forces every memory store + context technique to earn its place · Claude-native

THE PROJECT Why this one teaches everything

A long-running personal assistant you talk to across many sessions. It remembers facts, recalls your notes semantically, knows how people/projects relate, and manages long conversations. Each capability maps to a distinct, justified store — so you learn when to reach for each, not just how. The dialogue itself is where every context-bloat technique lives.

Deliberately toy & light/embedded (SQLite + Chroma + Kùzu in-process; Redis via brew or fakeredis) so you learn the concepts and APIs with almost no RAM/CPU cost — no Docker/Neo4j/Postgres tax.

WHAT IS THIS? The problem it solves

It's a personal work assistant — a "chief of staff" / second brain for one knowledge worker (you). A plain chatbot forgets everything when a session ends, so you re-explain your projects, people, and preferences every time. This assistant gives you continuity: it remembers your facts & preferences, recalls your past notes, tracks the people/projects you work with and how they connect, and keeps long conversations coherent. The value is recall across time — not a smarter single answer.

Not customer-facing — a private companion that keeps your working context so you never start from zero. The same shape generalises (a support agent = per-customer memory; a research assistant = a knowledge base), but here the "user" is one person, over months.

PART C Memory implementation — per store

Redis — working/session memory + cache D5
  • Keys: session:{id}:messages (recent turns), session:{id}:scratch, cache:{hash(query+memory_ver)}.
  • TTL on everything ephemeral (session expires; cache expires) — teaches eviction.
  • Also: a per-user rate limiter and the response/semantic cache (skip the LLM for repeats).
  • Failure: Redis down → degrade to stateless (rebuild session from durable stores); never hard-fail.
SQLite — structured facts & preferences
  • Tables: facts(user_id, key, value, confidence, source, updated_at), preferences(...).
  • Upsert by key so a changed preference overwrites (latest-wins) instead of accumulating contradictions.
  • Retrieval is exact/SQL — cheap, deterministic; great for "what's my timezone / default project".
  • Store provenance + timestamp + confidence so you can trust, age, and audit a fact.
Vector (Chroma) — episodic / semantic recall RAG
  • Write each note / meaningful chat snippet / doc chunk → embed → store with metadata (user, timestamp, type).
  • Recall by similarity to the question; cap to top-K with a score threshold (below it, retrieve nothing).
  • Chunk size/overlap matter; dedup near-identical memories; this is memory-as-RAG.
  • The raw episodic record here is also your safety net for compaction (see Part D): summarise the dialogue, but the detail still lives in the vector store.
Graph (Kùzu) — relationships & multi-hop
  • Nodes: Person, Project, Task; edges: WORKS_ON, REPORTS_TO, MENTIONED_IN.
  • Answers "who works with Priya on ABCCompany?", "what's blocked on the demo?" — questions a vector store can't (they need traversal, not similarity).
  • Extract entities/links from conversations (an LLM extraction step) and MERGE by a canonical id to avoid duplicate nodes.
  • Return a small k-hop neighbourhood, summarised — never the whole graph.

PART D Context engineering — the heart of it

LeverWhat you buildClaude / CCA
Memory routingA cheap classifier (rules or Haiku) decides which stores a turn needs — or query all in parallel and fuse. Skipping stores saves tokens & latency.D1/D5
What to remember (write policy)A "reflect" step after each turn extracts durable facts/entities and routes them to the right store — and skips the ephemeral chatter. Not everything is worth remembering.D3
Retrieval fusionMerge results from 4 stores into ONE small memory context block with a per-store token budget (e.g. top-3 vector, N facts, a summarised graph neighbourhood). Place it AFTER the cached prefix.D5
Prompt cachingCache the stable persona + instructions + tool defs (byte-stable prefix); the volatile memory block & user turn go after the last breakpoint.prompt caching D5
Context editingTool calls (web fetch, doc read) return big payloads → clear stale tool_results while keeping the dialogue.context editing D5
CompactionWhen a single long session nears the window, summarise the dialogue (append response.content!). The detail is safe in the vector store.compaction D5
Safe truncationFallback sliding window that never orphans tool_use/tool_result pairs.D5
Golden rule of the design: durable knowledge lives in the stores; the context window holds only a small, fused, per-turn slice. The window is a workspace, not a filing cabinet.

PART F Cross-cutting design

ConcernHere
Security & privacyPer-user memory isolation, PII redaction on write, encryption at rest, a "forget me" path (GDPR-style delete across all four stores).
CostPrompt-cache the persona; Haiku for routing/extraction; dedup + decay memories; response cache; bounded per-store retrieval budget.
LatencyParallel store reads; route to skip stores; write memories asynchronously after the reply; cache.
ReliabilityAny store down → degrade (skip that memory tier, don't fail the turn); retries/backoff on the API; loop termination.
ObservabilityTrace each turn: which stores hit, what was retrieved & injected, tokens (incl. cache_read), what was written, latency per store.
Evals"Did it remember correctly?" — a golden set of (setup writes → later question → expected recall); measure recall precision, wrong-memory injection rate, and answer quality.
