🧱 NL→SQL on Databricks Unity Catalog

From the P0 toy to a real copilot · every answer mapped to a Claude feature & CCA domain

THE JUMP Toy → real: the problem changes shape

The P0 Claude API agent had one DuckDB with a handful of tables — you could paste the whole schema into the prompt. On Databricks Unity Catalog you have many catalogs → schemas → tables/views/materialized views, possibly thousands of columns and rows of metadata. You cannot send it all. The whole game becomes: get the right context to the model, keep it small, and prove the SQL is correct before you trust it. Every technique below is a Claude/CCA lesson.

Databricks ships its own NL→SQL (AI/BI Genie). We build our own on the Claude API so each piece maps to an exam concept — that's the point of this page.

PART A Grounding — hitting the right catalog·schema·table

Q: How do we ensure it forms SQL against the RIGHT catalog / schema / table?

You never let the model guess from thousands of tables. You give it a retrieval + grounding layer and make it look things up with tools instead of inventing names. Three moves:

  1. Retrieve candidate tables — at ingestion, pull every table's fully-qualified name + description + column list from UC (system.information_schema, DESCRIBE TABLE EXTENDED), embed it, and index it (Databricks Vector Search or pgvector). At query time, retrieve the top-K tables relevant to the question. RAG D5
  2. Ground with tools, not the prompt — expose search_tables(query) and describe_table(full_name). The model must call describe_table to see real columns before writing SQL. It queries reality → it can't hallucinate a column. tool use D4
  3. Enforce the 3-level namespace + an allow-set — require catalog.schema.table in generated SQL; reject any table not in the retrieved candidate set. Set a default catalog/schema in the cached system prompt to cut ambiguity. guardrail D1/D4
Governance is the real backstop. Run the agent under a service principal with least-privilege SELECT grants. UC enforces access at the engine — the model literally cannot read a table the principal can't, and row/column masks apply automatically. Same rule as P0: security lives at the grant layer, never in the prompt.

Q: How do we limit context — we can't send it all?
  • Retrieval (top-K tables) is the primary lever — send 3–8 candidate tables, not 3,000. RAG
  • Progressive disclosure — send names + one-line comments first; the model calls describe_table only for the tables it actually wants. Keeps the fixed prompt tiny. tool use
  • A curated semantic layer beats raw introspection — a human-maintained data dictionary / glossary (metrics, joins, business terms) is fewer tokens and higher quality. This is exactly what a Skill or a cached instruction doc is for (Part C). Skills D2
  • Prompt caching — cache the stable prefix (SQL dialect rules + safety rules + semantic glossary); put the per-question retrieved schema after the last cache breakpoint (it's volatile). prompt caching D5
  • Tool search / defer_loading if you expose many per-domain tools. tool search D4
  • Context editing / compaction for long multi-question sessions — clear stale tool results so the window stays lean. context editing D5

PART B Correctness — syntactic, right-tables, and logically right

Q: How do we ensure the SQL is syntactically valid, against the right tables, and logically correct?

Three different problems, three different guards. Never trust the model's SQL directly.

Notice: most of this is a deterministic workflow (retrieve → generate → validate → execute → summarize). The only genuinely agentic part is the fix-on-error loop. That's the D1 lesson — don't agentify the parts you can hardcode.
Q: What test harness do we build?

A golden-query eval suite across catalogs — the thing that tells you it works and catches regressions when the schema, prompt, or model changes. evals D5

MetricWhat it measures
Execution accuracyDoes the generated SQL run without error?
Result-set matchDoes it return the gold answer (order-insensitive)? — the real "correct"
Table-selection precision/recallDid it pick the right tables from the metastore?
Self-correction countHow many EXPLAIN retries to get valid SQL (efficiency signal)

Refusal / clarify rateHow often it declines or asks — calibrate ambiguity handling
  • Golden set = (question, gold SQL, gold result) pairs, incl. ambiguous and adversarial questions.
  • For fuzzy answers, use an LLM-as-judge with a different model + a rubric. D5
  • Run against a staging metastore snapshot; re-run on every prompt/model/schema change (regression gate).
  • Track drift — execution accuracy quietly dropping as the schema evolves is your early warning.
Q: How do we bring hallucination down?
  • Ground, don't guess — the model must describe_table real columns before writing SQL. This alone kills most column hallucination.
  • EXPLAIN rejects any hallucinated table/column (they don't resolve).
  • Constrain to the retrieved allow-set — no querying random tables.
  • Give it an out — an ask_user() tool so it clarifies instead of inventing an answer to an ambiguous question. D3
  • Rich comments + sample values so business terms map to real columns/codes.
  • Cite which tables/columns it used, and show the SQL — the user can sanity-check.

PART C The agent — pattern, tools, CLAUDE.md & Skills, error handling

Q: What agentic pattern fits here?

✅ Default: single agent + tools (hub-and-spoke)

  • One agent, tools: search_tables · describe_table · validate_sql · run_sql · ask_user
  • The self-correcting loop over EXPLAIN/execute errors
  • Wraps a deterministic retrieve→validate→execute pipeline
  • Right for ~90% of NL→SQL (D1: single-agent-plus-tools beats multi-agent here)

⚙️ Reach further only when warranted

  • Fresh-context verifier sub-agent to double-check the SQL for high-stakes queries (D1 sub-agent isolation)
  • Planner + parallel sub-agents only for genuinely multi-part analytical questions (decompose → sub-queries → merge)
  • Programmatic tool calling to chain many describe_table calls in code — only the final result returns to context (token saver, D4)
Q: What tools do we expose? D4 · Tool Design
ToolDoesGuardrails / notes

Tool-design rules that are exam gold: the description is the #1 lever (say when to use it), constrain inputs with input_schema (+ strict), return failures as tool_result with is_error:true so Claude recovers, and namespace tools if you have per-domain sets.

# The two load-bearing tools (abridged) — validate BEFORE you execute
def validate_sql(query):                 # deterministic guard: EXPLAIN parses+analyzes, no data read
    try:
        warehouse.execute(f"EXPLAIN {query}")   # errors on bad syntax OR unresolved table/column
        return {"ok": True}
    except Exception as e:
        return {"ok": False, "error": str(e)}   # fed back as is_error → self-correction (D1)

def run_sql(query):                      # read-only SP + LIMIT + timeout enforced at the engine
    return warehouse.execute(f"SELECT * FROM ({query}) LIMIT 200", timeout_s=30)
Q: How do CLAUDE.md and Skills fit? (exam D2)

Two honest interpretations — know both for the exam:

  • CLAUDE.md — a Claude Code (D2) feature used while you build the agent: it lives in the repo and guides Claude Code with conventions, how to run the eval harness, the UC connection setup, "never generate non-SELECT." Its runtime analog in the Messages API is your cached system prompt: Spark/Databricks SQL dialect, the 3-level FQN rule, SELECT-only + always-LIMIT, prefer PK/FK joins, and the semantic glossary. Same idea — a stable, cached instruction doc. D2 prompt caching
  • Agent Skills (SKILL.md) package reusable expertise, model-invoked via progressive disclosure (loads only when relevant → saves context). Build a databricks-uc-sql skill: dialect rules, common patterns (windowing, date filters, region-code mapping), a checklist, plus bundled scripts (an EXPLAIN validator, a formatter). On the Messages API, skills load via the container + code-execution + skills betas. Skills D2
Skills vs tools vs RAG (a classic exam distinction): Skills = reusable how-to knowledge (SQL patterns); tools = actions (describe/validate/run); retrieval = the specific schema data for this question. Three different jobs.
Q: Error handling? D4/D5
  • Tool errors → tool_result with is_error:true (EXPLAIN failure, run error, permission denied, timeout) → the model reads it and fixes.
  • API errors → SDK auto-retries 429/5xx with backoff; handle all six stop_reasons (tool_use / end_turn / max_tokens / refusal / stop_sequence / pause_turn) — see the Claude API build; never fall through.
  • Guardrail denials (non-SELECT, disallowed table) → return a clear message, don't execute.
  • Loop termination — cap self-correction turns; on exhaustion, return the best SQL for a human to run (graceful degradation).

PART D Production — cost, reliability, scale, observability

Q: How do we bring the cost down? D5
  • Prompt-cache the stable system prompt + SQL guidelines + glossary (~90% cheaper cached reads; put volatile retrieved-schema after the last breakpoint).
  • Model tiering — Haiku 4.5 for retrieval/routing & simple SQL; Opus/Sonnet for hard multi-join reasoning. Route by question complexity.
  • Retrieval keeps input tokens tiny (don't send the metastore).
  • Cache introspected metadata + sample values — refresh only on DDL change, not per request.
  • EXPLAIN before run — avoid paying warehouse compute for a wrong query (Databricks cost, and it's the bigger bill).
  • Batch API (~50% off) for scheduled/offline report generation; effort low for simple SQL.
  • Bound self-correction retries (each is a full request).
Q: How do we make it reliable? D5/D1
  • Deterministic validation (EXPLAIN) + bounded self-correction — the core reliability move.
  • Retries w/ backoff on 429/5xx (SDK); fallbacks — model fallback on overload, and a degraded mode that returns the SQL for a human if execution keeps failing.
  • Warehouse timeouts + statement cancellation; a circuit breaker if the warehouse is down.
  • Idempotency is free — read-only queries are naturally safe to retry.
  • Least-privilege SP + LIMIT + row cap bound the blast radius.
Q: How do we make it scalable?
  • Stateless service → scale horizontally (many workers); prompt cache is shared.
  • Vector index scales to many tables; refresh incrementally on DDL/lineage changes.
  • Watch ITPM/OTPM rate limits — queue, and spread warehouse concurrency (serverless SQL, multiple warehouses).
  • Async / Batch for heavy or scheduled loads; per-tenant isolation via the SP's grants (multi-catalog/workspace).
Q: What observability do we build? D5
  • Trace every request: question → retrieved tables → generated SQL → EXPLAIN result → #retries → final SQL → rows → latency → tokens (input/cache_read/cache_creation/output) → cost → warehouse bytes scanned → success/fail. Use MLflow Tracing / Langfuse / Phoenix.
  • Usage fields → cost per query + cache hit-rate; capture request-id/_request_id for support.
  • SLOs: valid-SQL rate, execution accuracy (from evals), p50/p95 latency, cost/query, cache hit-rate, retry rate, refusal rate.
  • Databricks audit — query history / system tables record the executed SQL; lineage shows what was touched.
  • Alert on error-rate/cost spikes and drift (execution accuracy falling as the schema changes).
