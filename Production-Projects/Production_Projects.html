Project 1: Simple Chatbot

Duration: 2 weeks
Domains: D1, D3
Tech: Flask + Claude API

Goal

Build a conversational chatbot that demonstrates basic architecture patterns and prompt engineering.

What You'll Learn

  • How Claude API works (completions, streaming, system prompts)
  • Token management (conversation history pruning)
  • Session persistence (SQLite or in-memory)
  • Basic error handling & retries
  • Prompt engineering for specific tone/voice
  • Measuring performance (latency, cost)

Implementation Checklist

Success Criteria

Responds in <2 seconds
Costs <$0.01 per request
Handles 100 turns before context overflow
Gracefully handles errors

Project 2: Customer Support Agent

Duration: 3 weeks
Domains: D1, D4, D5
Tech: FastAPI + SQLite

Goal

Build a real agent that uses tools to solve customer issues without manual escalation (or escalates intelligently).

Tools to Implement

  1. query_ticket_history(customer_id) – Recent tickets
  2. get_customer(id) – Account status, plan, billing
  3. search_knowledge_base(query) – Relevant docs, FAQs
  4. create_escalation(issue_type, priority) – Hand off to human
  5. log_resolution(ticket_id, solution, tags) – Track for analytics

What You'll Learn

  • Agentic loops: request -> check stop_reason -> call tool -> repeat
  • Tool descriptions (more important than function names!)
  • Error handling: what if a tool fails?
  • Context management: when to include customer history vs. omit
  • When to escalate (agent confidence, complexity, security)
  • Measuring agent performance (resolution rate, cost, time)

Success Criteria

Resolves 70%+ without escalation
Latency <5 seconds
Cost <$0.05 per ticket
Correct escalations (no billing handled, no password resets)

Project 3: RAG System

Duration: 3 weeks
Domains: D4, D5
Tech: Chroma + Embeddings

Goal

Build a retrieval-augmented generation system for a document corpus. Learn why retrieval strategy matters.

Pipeline

Query -> Embed -> Vector search (top-10) -> Format -> Claude -> Answer + Citations

What You'll Learn

  • Chunking strategies (fixed size, semantic, recursive)
  • Embedding models (OpenAI, local, specialized)
  • Vector database setup (Pinecone, Chroma, Qdrant)
  • Retrieval ranking: keyword + semantic hybrid?
  • Reranking: simple heuristics vs. LLM-based
  • Context formatting & token budgeting
  • Evaluation: ROUGE, MRR, human QA

Key Experiment

Test 3 chunking strategies on your corpus:

  1. Fixed 256-token chunks (baseline)
  2. Semantic chunking (topic-based)
  3. Recursive chunking (hierarchical)

Measure: retrieval quality, latency, cost. Which is best?

Success Criteria

Retrieval precision (top-5 results relevant) >80%
Answer accuracy (vs. eval set) >70%
Latency <3 seconds
Cost per query <$0.10
Can cite sources

Project 4: Multi-Agent Workflow

Duration: 3 weeks
Domains: D1, D2
Tech: FastAPI + Multiple Claude instances

Goal

Orchestrate multiple specialized agents working together. Learn when to use multi-agent vs. single agent + tools.

Scenario: Content Creation Workflow

User: "Write a blog post about React hooks"
    |
Coordinator Agent (decides workflow)
    |-- Research Agent (searches, summarizes)
    |-- Outline Agent (structures)
    |-- Draft Agent (writes)
    |-- Editor Agent (checks grammar)
    |-- Reviewer Agent (fact-checks)
    |
Final Blog Post

What You'll Learn

  • Coordinator pattern: central agent routes work
  • State management: passing context without explosion
  • Tool allocation: which agent gets which tools?
  • Error handling: if one agent fails, what then?
  • Prompt isolation: agents don't know about each other
  • Serial vs. parallel: latency trade-offs

Success Criteria

All agents coordinate without error
Final output is coherent
Latency <30 seconds (for 5-agent workflow)
Can explain: why multi-agent here vs. single+tools?

Project 7: Enterprise RAG (Advanced)

Duration: 4 weeks
Domains: D1, D4, D6
Tech: Hybrid retrieval + Caching

Goal

Scale Project 3 (RAG) to production. Learn what happens when RAG hits real constraints.

Production Constraints

  • Documents: 100k+, updated hourly
  • Queries: 1000+/day
  • Latency: <3 seconds SLA
  • Cost: $1000/month budget
  • Quality: >80% answer relevance

Architecture

[1] Keyword search (BM25) -> 1000 results
[2] Semantic search -> 1000 results
[3] Merge & dedupe -> 500 results
[4] Time filter (7 days) -> 200 results
[5] Rerank (LLM) -> top-10
[6] Format with citations
[7] Claude (with prompt cache) -> Answer

What You'll Learn

  • Hybrid retrieval: keyword + semantic
  • Time-aware retrieval: filtering by recency
  • Multi-stage retrieval: coarse -> fine -> rerank
  • Prompt caching: ROI analysis
  • Cost optimization: Opus vs. Sonnet
  • Production evaluation: automated + human evals

Success Criteria

Retrieval quality >80%
Latency <3 seconds end-to-end
Cost <$1 per query
System handles 100 concurrent
Explain: why hybrid? Why that reranker?

Project 8: Security & Compliance

Duration: 2 weeks
Domains: D5
Tech: Encryption + Audit Logging

Goal

Harden one of the previous systems for regulated industry. Learn compliance constraints.

Scenario: HIPAA-Compliant Healthcare Agent

Your customer support agent now handles healthcare data. What changes?

Security Measures

  • Encryption: At rest (database) + in transit (TLS)
  • PII Handling: Redaction before storage
  • Access Control: Only authorized staff see conversations
  • Audit Logging: Who accessed what, when, why
  • Data Retention: Automatic deletion after period

What You'll Learn

  • Data security: encryption, access controls
  • Compliance frameworks: SOC2, HIPAA, GDPR
  • Audit logging: what, how long, tamper-proof
  • Threat modeling: what could go wrong?
  • Design constraints from compliance

Success Criteria

All sensitive data encrypted
Audit log is tamper-proof
PII redacted before storage
Can demonstrate compliance
<100ms latency impact from security

Implementation Tips

Document Your Decisions

For each project, write a 1-page Architecture Decision Record (ADR):

## Project X: Decision Y

### Context
Why did we need to decide this?

### Decision
What did we choose?

### Rationale
Why this option over alternatives?

### Trade-offs
What are we giving up?

### Cost Impact
What's the latency/cost/complexity impact?

Use Projects as Exam Case Studies

In the exam, you'll see scenarios. You've literally built this. You can reference:

  • "I built a similar system. Here's what worked..."
  • "Compliance required X, which added Y latency..."
  • "Cost per request came to $Z, fitting within budget..."
