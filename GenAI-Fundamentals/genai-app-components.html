🧩 Anatomy of a Gen AI Application

The major components of a real product — and how they fit together

BIG PICTURE An LLM alone is not an application

A raw model just turns tokens into tokens. A production Gen AI application wraps that model in a set of components that give it your data, memory, tools, guardrails, and reliability. Almost every serious app — a support bot, a coding agent, a research assistant — is built from the same handful of building blocks.

TRADE-OFFS Three ways to give a model your knowledge

A recurring architecture decision: how do you get the model to know things it wasn't trained on? Three levers, increasing in weight:

ApproachHowBest forCost / effort
Prompting / in-contextPut the info directly in the promptSmall, per-request context🟢 Lowest
RAGRetrieve relevant chunks from a vector store at query timeLarge, changing knowledge bases🟡 Medium
Fine-tuningFurther-train the model weights on your dataFixed style/format or narrow domain🔴 Highest
Rule of thumb: start with prompting, add RAG when knowledge is large or changes often, and reach for fine-tuning only when prompting + RAG can't get the behavior you need.

DATA FLOW Two phases: ingestion vs. query

RAG has two separate timelines that beginners often conflate:

🗂️ Ingestion (offline, ahead of time)

  • Load source docs
  • Chunk into passages
  • Embed each chunk → vectors
  • Store in the vector DB

⚡ Query (online, per request)

  • Embed the user's question
  • Search the vector DB for nearest chunks
  • Inject chunks into the prompt
  • Model answers, grounded in them

TAKEAWAYS

Remember: the LLM is one component, not the whole app — most engineering effort goes into the layers around it.
Remember: RAG ≠ fine-tuning. RAG adds knowledge at query time without changing the model; fine-tuning changes the weights.
Remember: tools/MCP give the model actions & live data; RAG gives it relevant documents; memory gives it continuity. Different jobs.
