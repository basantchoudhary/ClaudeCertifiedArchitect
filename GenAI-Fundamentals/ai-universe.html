🌌 The AI Universe — Who Does What

One map of the whole landscape · concepts + the real players that fill each slot

BIG PICTURE The problem: everyone seems to do everything

Anthropic, OpenAI, Google, Databricks, Azure, Pinecone, LangChain, MCP… the names blur together because each one lives on a different layer of the same stack — and some sit on several layers at once. The trick to seeing clearly is to stop asking "what is this company?" and start asking "which layer is this thing on?". There are really only six layers, plus a set of cross-cutting concerns.

🔑 Mental model: a Gen AI system is a stack. At the bottom is raw compute; at the top is the app a human uses. Every product name you've heard slots into one (or more) of these layers.

KEY IDEA One player, many hats

The confusion comes from companies that operate on several layers at once. That's normal — it's vertical integration. Here's how the big names spread across the stack:

🟣 Anthropic (a lab, integrated upward)

  • Layer 2 — the lab that trains the models
  • Layer 3 — Claude is the model family
  • Layer 4 — the Anthropic API (Messages API)
  • Layer 5 — MCP (the protocol) + the Agent SDK
  • Layer 6 — Claude Code & the Claude apps

🔵 Microsoft / Azure (a cloud, integrated downward)

  • Layer 1 — Azure cloud + GPUs
  • Layer 4 — Azure AI Foundry hosts many models (OpenAI, Anthropic, Meta, …)
  • Layer 5/6 — Copilot Studio, GitHub Copilot
  • Note: Microsoft doesn't (mostly) train frontier models — it hosts and packages others'
So "Claude" (a model, layer 3) can be reached through the Anthropic API (layer 4) or through AWS Bedrock, Google Vertex AI, Azure AI Foundry, or Databricks — all layer-4 gateways. Same model, different front door. That's why the same name shows up in many places.

  1. Prompting — the app + orchestrator write the instructions and assemble context.
  2. RAG — the query is turned into an embedding, matched against a vector store, and the top chunks are pulled into the prompt so the model answers from your data.
  3. Caching — a stable prompt prefix is reused to cut cost & latency; memory carries facts across turns/sessions.
  4. Inference — the LLM runs (through an API/gateway) and generates a response.
  5. Tools / MCP — if it needs live data or actions, it calls tools, often over MCP, then continues.

GLOSSARY The acronyms, fast

TermIn one sentence
LLMLarge Language Model — a neural net trained to predict the next token; the "brain."
Foundation modelA large, general model trained on broad data, adaptable to many tasks (LLMs are one kind).
TokenA chunk of text (~¾ of a word) — the unit models read, generate, and are billed on.
Context windowThe max tokens a model can consider at once (input + output).
InferenceRunning a trained model to produce output (vs. training, which creates it).
EmbeddingA vector of numbers representing meaning, so similar text sits close together.
Vector storeA database that stores embeddings and finds the most similar ones fast.
RAGRetrieval-Augmented Generation — fetch relevant data and feed it to the model at query time.
PromptThe instructions + context you send the model to steer its output.
Fine-tuningFurther-training a model on your data to change its behavior (heavier than prompting/RAG).
AgentAn LLM that decides and acts in a loop, calling tools until a task is done.
MCPModel Context Protocol — an open standard for plugging tools & data into any model/app.
Tool / function callingLetting the model invoke your code (search, DB, API) and use the result.
Prompt cachingReusing a processed prompt prefix to cut cost & latency on repeated requests.
