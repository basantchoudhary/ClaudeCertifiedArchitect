7.5 · Subagents vs Skills vs Commands

When to reach for each

CONCEPT Three ways to package behaviour — one axis separates them

Subagents, skills, and slash commands all extend Claude Code, but they differ on two questions: who invokes it and which context window it runs in.

MechanismInvoked byRuns inAdds
SubagentClaude (auto-delegate) or you (explicit)Its own isolated context windowContext isolation + tool restriction + optionally a different model
SkillClaude (model-invoked)The same (main) context windowReusable prompts / workflows / instructions in-context
Slash commandYou (typed /name)The same (main) context windowA user-triggered, reusable prompt
🔑 The distinguishing feature of a subagent is context isolation — plus the ability to restrict tools and pick a different model. Skills and commands run in the same context as your conversation.

CONCEPT Choosing between them

The official guidance is direct: consider a Skill instead of a subagent when you want reusable prompts or workflows that run in the main conversation context rather than in an isolated subagent context. So:

  • Reach for a subagent when the work is verbose or self-contained and you want it out of the main window, or when you need to restrict tools / route to a cheaper model.
  • Reach for a skill when you want a reusable capability Claude invokes on its own, but the work should stay in the main conversation with full shared context.
  • Reach for a slash command when you want to trigger a reusable prompt on demand by typing /name.
Rule of thumb: need a clean context window and/or tool & model control → subagent. Need a model-invoked capability that shares context → skill. Need a user-triggered prompt → command.

CODE Same idea, three packagings

A "code review" capability, expressed each way — note where each lives and who fires it.

# SUBAGENT — .claude/agents/code-reviewer.md
# Claude delegates when the task matches; runs in its OWN context.
---
name: code-reviewer
description: Expert code reviewer. Use proactively after code changes.
tools: Read, Grep, Glob
---
You are a senior code reviewer...

# SKILL — model-invoked, runs in the MAIN context (shares your conversation)
# SLASH COMMAND — you type /review to fire a reusable prompt in the MAIN context
⚠️ Same wording ("review my code") can trigger different mechanisms. The material difference is where it runs: a subagent isolates it; a skill and a command keep it in your conversation.

TRADE-OFFS Isolated subagent vs. in-context skill/command

✅ Subagent (isolated)

  • Keeps verbose work out of the main window
  • Can restrict tools and pin a different model
  • Independent permissions per agent

⚠️ Skill / command (shared context)

  • Full shared context — no re-explaining the situation
  • Lower latency; no fresh-start context gathering
  • But verbose output lands in your main conversation

EXAM TRAPS

Trap: "Skills and subagents are the same thing." → A skill is model-invoked and runs in the same context; a subagent runs in an isolated context window and adds tool restriction and optional model choice. Different context is the key distinction.
Trap: "Slash commands and subagents are interchangeable." → A slash command is a user-invoked prompt that runs in the main context. A subagent is delegated (auto or explicit) and runs in its own context.
Trap: "Use a subagent for any reusable prompt." → If you just want a reusable prompt or workflow in the main conversation context, the docs point you to a Skill instead. Subagents are for isolation / tool / model control.
Trap: "Only subagents can restrict tools or change the model." → That combination — context isolation + tool restriction + optionally a different model — is exactly what a subagent adds over a skill or command. That's the reason to pick one.

PRACTICE

Q1 · What is the defining difference between a subagent and a skill?

A subagent runs in its own isolated context window (and can restrict tools / use a different model); a skill is model-invoked and runs in the same context as the main conversation.

Q2 · You want a reusable workflow that stays in the main conversation. Which mechanism?

A Skill — the docs specifically recommend a skill over a subagent when you want reusable prompts/workflows in the main context rather than isolated subagent context.

Q3 · What does a subagent add over a slash command?

Context isolation (its own window), tool restriction, and optionally a different model. A slash command is a user-invoked prompt that runs in the main context with no isolation.
