1.2 · Token Counting & the Count Endpoint

Measure before you send

CONCEPT Measure a prompt before it hits the window

You can find out exactly how many input tokens a request will cost before sending it, using the token counting endpoint: POST /v1/messages/count_tokens. It accepts the same structured inputs as message creation — system, messages, tools, images, and PDFs — and returns the total input tokens as a single field:

{ "input_tokens": 14 }

Use it to proactively manage rate limits and cost, make model-routing decisions, and fit a prompt to a target length — and, most relevant here, to budget against the context window so a request doesn't overflow (1.5).

🔑 The count is an estimate — the actual number used when creating the message may differ by a small amount. Anthropic may also add tokens for system optimizations, but you are not billed for system-added tokens — billing reflects only your content.

CONCEPT Model-specific, free, and separately rate-limited

Token counts are model-specific — pass the same model you'll use for inference. This matters: Opus 4.7+, Fable 5, Mythos 5, and Sonnet 5 use a newer tokenizer that produces roughly 30% more tokens for the same text than earlier models. Never reuse a count measured against an older model to estimate context-window fit — recount against the target model.

CODE Budget-check before creating the message

# Free, and doesn't touch your message-creation rate limit.
count = client.messages.count_tokens(
    model="claude-opus-4-8",   # same model you'll infer with — tokenizer differs!
    system="You are a scientist",
    tools=tools,                    # tool defs count too
    messages=history,
)
input_tokens = count.input_tokens   # e.g. 403

WINDOW, MAX_OUT = 1_000_000, 4096
if input_tokens + MAX_OUT > WINDOW:
    history = summarize_oldest(history)   # make room BEFORE sending (1.5)

resp = client.messages.create(
    model="claude-opus-4-8", max_tokens=MAX_OUT, messages=history,
)
To size a diff, the endpoint is stateless: count each version separately and subtract. Server-tool token counts apply only to the first sampling call.

TRADE-OFFS Pre-count vs. send-and-see

✅ Count first

  • Catch overflow before spending a create call
  • Free; own rate-limit pool
  • Drives model routing & cost estimates

⚠️ Costs / caveats

  • An extra round trip & its own RPM limit
  • Only an estimate — small drift vs. actual
  • Doesn't use caching logic (no cache benefit)

EXAM TRAPS

Trap: "Use tiktoken / a fixed chars-per-token ratio." → Wrong tokenizer, wrong numbers. Use POST /v1/messages/count_tokens against the target model.
Trap: "A count from Sonnet 4.5 is fine for Opus 4.8." → The newer tokenizer (Opus 4.7+, Fable 5, Sonnet 5) yields ~30% more tokens for the same text. Recount per model.
Trap: "Counting eats my message rate limit." → No — token counting has separate, independent rate limits, and the endpoint is free.
Trap: "The count is exact." → It's an estimate; actual may differ by a small amount. Anthropic's system-added tokens aren't billed.

PRACTICE

Q1 · What endpoint estimates a prompt's size, and what does it return?

POST /v1/messages/count_tokens (SDK: messages.count_tokens). It takes the same inputs as message creation and returns { "input_tokens": N } — an estimate of total input tokens including system, messages, tools, images, and PDFs.

Q2 · Why must you pass the exact model you'll infer with?

Token counts are model-specific. Opus 4.7+, Fable 5, Mythos 5, and Sonnet 5 use a newer tokenizer that produces ~30% more tokens for the same text — a count from an older model would under-estimate context-window fit and cost.

Q3 · Is counting free, and does it consume your message rate limit?

Counting is free, and it has separate, independent RPM limits (Start 2,000 / Build 4,000 / Scale 8,000) that don't count against message creation.
