🧯 Who Catches What — 400 vs refusal vs is_error

Server-side validation vs a model decision vs your app · and every reason a request 400s

THE ANSWER A 400 is caught before the model runs

An invalid request (HTTP 400, invalid_request_error) is caught by Anthropic's API server at the validation layer — before any model inference. The LLM never sees a malformed request, and you're not billed any tokens for a 400. It's deterministic schema/structure/parameter checking, not the model "noticing" a problem.

So your example — a tool_result in a user turn with no preceding assistant tool_use, or an assistant tool_use not followed by its tool_result — is a structural contract violation caught server-side as 400, never handled by the LLM.

PIPELINE Where each error lives

A request passes through stages; a failure at each returns a different code. The model only runs if validation passes.

WHO DECIDES 400 vs refusal vs is_error vs max_tokens

The distinction people get wrong: a structure/parameter problem is a 400 (server); a behavioural outcome is a successful call (HTTP 200) with a stop_reason the model chose.

SignalCaught byHTTPBilled?Example
Invalid request (400)Server — before the model400 errorNomalformed messages, bad params, broken tool pairing

A refusal is NOT a 400. It's a successful HTTP 200 the model chose to decline — always check stop_reason before reading content. And is_error:true is you reporting a tool failure into the conversation, not an API error at all.

400 CAUSES Every reason a request raises 400

A · Message structure & the tool contract

  • messages roles don't alternate user/assistant
  • first message is assistant (must be user)
  • empty messages array
  • a tool_result with no matching tool_use in the preceding assistant turn
  • an assistant tool_use not immediately followed by its tool_result (literal error: "tool_use ids were found without tool_result blocks immediately after")
  • text/blocks interleaved incorrectly around tool blocks

B · Malformed / missing basics

  • malformed JSON body
  • missing required params (model, max_tokens, messages)
  • wrong parameter types (string where an int is expected)
  • invalid tool-definition schema

C · Parameter constraint violations

  • max_tokens exceeds the model's output limit
  • budget_tokens >= max_tokens (on models that still use extended thinking)
  • out-of-range values (e.g. invalid temperature)

D · Model-specific 400s (current models — Opus 4.8/4.7, Sonnet 5, Fable 5)

  • sending temperature / top_p / top_k at all — removed → 400
  • thinking:{type:"enabled", budget_tokens:N} — removed → 400 (use {type:"adaptive"})
  • assistant-turn prefill (last message is assistant) → 400 on the 4.6/4.7/4.8/Fable-5 family
  • Fable 5 only: explicit thinking:{type:"disabled"} → 400 (omit the param instead)
  • citations combined with output_config.format (structured outputs) → 400
  • Fable 5 under zero-data-retention / retention below 30 days → every request 400s (org config, not the payload)
413 request_too_large and 429 rate_limit_error are their own codes, not 400. 401/403 are auth/permission; 500/529 are server-side (retryable).

CONSEQUENCE Why SDKs don't retry a 400

Because a 400 is deterministic server-side validation, it never fixes itself. That's why the Anthropic SDKs auto-retry 429 and 5xx (transient) with backoff, but never retry 400 — resending the identical malformed request just 400s again. The fix is always in your request construction.

This is exactly why a correct agent loop appends the whole assistant turn and returns all tool_results in one user message — it satisfies the tool contract and avoids these 400s. See D4 · Tool-Result Contract Failures.

EXAM ONE-LINERS

• 400 = server-side validation, before the model, unbilled, deterministic — the LLM never sees an invalid request.
• refusal is a model decision returned as HTTP 200 with stop_reason:"refusal" — check stop_reason before reading content.
• is_error:true on a tool_result is your app reporting a tool failure to the model — not an API error.
• SDKs retry 429/5xx, never 400.
