🧹 Fixing Context Bloat — Which Technique, When

Pick the right lever · keep integrity & reliability intact

FRAME First ask: what kind of bloat is it?

"Context is too big" isn't one problem — it has several causes, and each has a different right fix. Diagnose the cause first, then apply the matching technique. Using the wrong one (e.g. summarising when you just needed to prune tool results) is how you break the conversation.

🔑 The two most-confused options: context editing PRUNES stale tool results (keeps the dialogue); compaction SUMMARISES the whole history (rewrites it). If the problem is specifically bloated tool_results and the running dialogue must stay intact → context editing, every time.

Compaction
compact_20260112		 The whole conversation is approaching the context-window limit Summarises earlier history into a compaction block, server-side MUST append response.content (not just .text) back to messages every turn — the compaction block carries the state; dropping it corrupts the next request
Memory tool
memory_20250818		 Facts must persist across sessions, not just within one conversation Claude reads/writes a /memories dir you back with storage Lives outside the window (no bloat). Guard against path traversal; never store secrets/PII (it replays into every future session)

(client-side sliding window)		 A simple "keep the last N turns" suffices; few/no tools Drops the oldest turns yourself Cut only at a plain user turn so a tool_use never loses its tool_result; keep the first (instructions) turn. It invalidates the prompt cache (prefix changes) — prefer context editing when you only need to drop tool results
Tool slimming
(design)		 Fat tool schemas or huge tool outputs are the bloat Tool search + defer_loading; return lean results; programmatic tool calling Doesn't change roles/pairing — just keeps each turn small. Tool search appends schemas, so the cache is preserved

CONTRAST The distinctions the exam probes

🟢 Context editing (prune)

  • Targets stale tool_results specifically
  • Dialogue & message structure untouched
  • Lossless for the conversation (only junk removed)
  • Use when: tool payloads bloat, dialogue must stay

🟠 Compaction (summarise)

  • Condenses the whole history into a summary
  • Lossy — the dialogue is rewritten
  • Needs the response.content round-trip or state is lost
  • Use when: the entire conversation nears the window limit
Trap: "context is too big" ⇒ reach for compaction. Not always — if it's tool-result bloat with a live dialogue, that's context editing. And prompt caching does not shrink the window at all — it only makes re-sending cheaper.

STACK They combine

Real long-running agents use several at once: prompt-cache the stable prefix, context-edit stale tool results as the loop runs, compact if the whole thing still approaches the limit, and use memory for anything that must outlive the session. Keep tool schemas/results lean throughout.
