Overview

An agent is a loop wrapped around one stateless endpoint (POST /v1/messages). You send the whole conversation; the model replies; if it asked for a tool, you run it, append the result, and call again — until it stops. Everything on this page is verified against the current Messages API, not just the summary study guide.

3 · Message structure & the tool-result contract

sequenceDiagram participant H as Your harness participant C as Claude (Messages API) H->>C: user turn — weather in Paris? C-->>H: assistant turn — tool_use (id = toolu_A) Note over H: run get_weather(Paris) → 18C sunny H->>C: user turn — tool_result (tool_use_id = toolu_A) ✅ ids match C-->>H: assistant turn — It's 18C and sunny. (end_turn) Note over H,C: results in a USER turn · ids MUST match · roles alternate
Break any → HTTP 400. Tool errored? add is_error = true
  • messages[] alternates user / assistant; first message must be user.
  • A tool_result is user-role content; its tool_use_id must match the tool_use block it answers.
  • Return all results from one assistant turn in a single following user message.
  • Failed tool → set "is_error": true so Claude adapts.

Golden rule: pick the simplest tier that works. "Should I build an agent at all?" — only if it's Complex, the Value justifies cost/latency, Claude is Viable at it, and errors are recoverable (Cost-of-error). Any "no" → simpler tier.

5 · Orchestration patterns

flowchart TD subgraph HS["Hub & Spoke — start here"] A1["🤖 agent"] --> T1["🛠 tool"]; A1 --> T2["🛠 tool"]; A1 --> T3["🛠 tool"] end subgraph CO["Coordinator / Sub-agent"] K["🧭 coordinator"] --> R1["🤖 refund"]; K --> R2["🤖 technical"]; K --> R3["🤖 billing"] R1 --> MG(["merge → response"]); R2 --> MG; R3 --> MG end subgraph SP["Sequential pipeline"] P1["research"] --> P2["outline"] --> P3["draft"] --> P4["edit"] end
🔑 Sub-agents are context-isolated — they don't see the coordinator's history. Pass what they need explicitly. Parallelise independent agents to cut latency; don't switch the main loop's model mid-conversation (it kills the prompt cache) — spawn a sub-agent instead.

8 · Code snippets

The manual loop, in miniature

# stateless: resend the full messages[] every call
messages = [{"role": "user", "content": user_input}]
while True:
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=16000,
        tools=TOOLS, messages=messages,
    )
    if resp.stop_reason == "end_turn": break
    if resp.stop_reason == "pause_turn":            # server-tool paused
        messages.append({"role": "assistant", "content": resp.content})
        continue                                    # re-send as-is; server resumes
    messages.append({"role": "assistant", "content": resp.content})  # WHOLE turn!
    results = []
    for b in resp.content:
        if b.type == "tool_use":
            results.append({"type": "tool_result",
                            "tool_use_id": b.id,          # MUST match, or 400
                            "content": run_tool(b.name, b.input)})
    messages.append({"role": "user", "content": results})  # results in a USER turn

10 · Practice questions

Q1 · Your loop appends only response.content[0].text. The 2nd tool call errors. Why?

You must append the entire response.content. The assistant turn holds the tool_use block; dropping it means the next tool_result references a tool_use_id not in history → 400.

Q2 · stop_reason == "max_tokens". What happened, best fix?

The output hit its cap and is truncated — not "conversation too long." Raise max_tokens, stream, or continue the turn. Summarising history is the wrong lever.

Q3 · A web-search call returns pause_turn. What do you send next?

Re-send the assistant turn as-is; the server resumes its own loop. Adding a "Continue." user message is wrong — the API detects the trailing server_tool_use block. Cap continuations to avoid infinite loops.

Q4 · 10 tools, one escalation path, cost- & latency-sensitive. Single or multi-agent?

Single agent + tools. No distinct roles or parallel work; cost + latency both favour the simplest tier. Multi-agent adds tokens/latency for no benefit.

Q5 · A billing sub-agent keeps re-asking for the customer ID the coordinator had. Cause?

Lost state / prompt isolation. Sub-agents don't share the coordinator's context — pass the required state explicitly in the delegated message.

Q6 · Code reads content[0].text and sometimes throws an index error. Why?

The turn was a refusal (or first block isn't text) — content can be empty. Branch on stop_reason first; for refusals inspect stop_details.

Q7 · You want a cheaper model for just one sub-step of a long Opus chat. Best approach?

Spawn a sub-agent on the cheaper model. Switching the main loop's model invalidates the model-scoped prompt cache, forcing a full re-process of the prefix.

11 · Exam traps & quick answers

Trap / keywordAnswer
"How does the agent remember earlier turns?"You resend full messages[]; the API is stateless.
"Loop appended reply but next tool call failed"Appended only .text; append the whole response.content.
"Server tool returned pause_turn"Re-send assistant turn unchanged; don't add "Continue."
"max_tokens — conversation too long?"No — the output hit its cap.
"Which stop_reasons exist?"end_turn, tool_use, max_tokens, stop_sequence, pause_turn, refusal.
"cost-effective"Single agent + tools · Sonnet/Haiku · lower effort.
"least latency"Parallelise; don't add agents.
"best-practice / robust"Explicit state, retries, circuit breakers.
