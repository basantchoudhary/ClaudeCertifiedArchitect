6.4 · Fallback Strategies

When retry won't help: alternate tool · skip · degraded path

CONCEPT Rung 2 of the recovery ladder

Retry (6.3) handles transient failures. When a failure is durable — the tool is down, the input is unsupported, retries exhausted — you climb to fallback. A fallback keeps the task moving on a lesser path instead of failing the whole request.

FallbackWhat it doesUse when
Alternate toolRoute to a second tool that serves the same needThe task still fully achievable another way (e.g. secondary search provider)
SkipOmit the failed step and proceedThe step was optional / enrichment, not load-bearing
Degraded pathReturn a partial or lower-fidelity resultA worse-but-useful answer beats no answer (e.g. cached data, coarser estimate)
🔑 The most important move is to tell Claude the tool failed. Return the tool_result with "is_error": true and a short reason. The model then adapts — tries a different tool, asks for clarification, or answers without that data — instead of hallucinating a result.

CODE Alternate-tool fallback + is_error signal

Try the primary tool; on durable failure try an alternate; if that also fails, hand the model an error result so it can adapt rather than fabricate.

def run_tool(name, args):
    # Primary → alternate → signal failure to the model.
    try:
        return {"content": primary_search(args["q"])}
    except Exception:
        try:
            return {"content": backup_search(args["q"])}   # alternate tool
        except Exception as e:
            # Degraded/skip: hand the model an ERROR so it adapts.
            return {"content": f"search unavailable: {e}", "is_error": True}

# In the agent loop, attach is_error to the tool_result block:
out = run_tool(block.name, block.input)
tool_result = {
    "type": "tool_result",
    "tool_use_id": block.id,
    "content": out["content"],
    "is_error": out.get("is_error", False),   # True → Claude sees the failure
}
messages.append({"role": "user", "content": [tool_result]})
By marking the block is_error: true, the next turn's model sees "this tool failed" and can pick another approach or explain the gap — turning a crash into graceful degradation.

TRADE-OFFS Fallback vs. failing loudly

✅ Fallback / degrade

  • Task still completes (partially) — better UX under partial outage
  • Contains blast radius of one flaky dependency

⚠️ Silent fallback risk

  • A degraded answer can look authoritative — mask the outage
  • Wrong for high-stakes steps (payments, legal) — those should escalate, not silently degrade
  • Needs surfacing: log + flag the reduced fidelity
⚠️ Never silently fall back on an irreversible or high-stakes action. There, the right rung is escalate to a human (6.5), not a quiet degraded path.

EXAM TRAPS

Trap: Retrying a durably-down tool forever. → After retries are exhausted, the ladder says fallback, not more retries. Retry is only for transient faults.
Trap: On tool failure, returning a fake/empty success. → Mark it is_error: true so the model adapts. A silent empty result invites hallucination.
Trap: Silently degrading a payment/irreversible step. → High-stakes failures escalate to a human (6.5); they don't get a quiet degraded path.

PRACTICE

Q1 · Your primary geocoding API returns 503 after retries. The task can be done by a second provider. Which fallback?

Alternate tool — route to the backup geocoder. The task is still fully achievable, so prefer the equivalent path before skipping or degrading.

Q2 · A tool fails and you can't recover its data. What must you return to the model, and why?

A tool_result with "is_error": true and a brief reason. It tells Claude the tool failed so it can adapt (different tool, ask the user, or answer without that data) instead of fabricating a value.

Q3 · When is a degraded path the wrong choice?

For high-stakes or irreversible actions (payments, deletions, legal commitments). A silent degraded result there can cause real harm and look authoritative — escalate to a human instead (6.5).
